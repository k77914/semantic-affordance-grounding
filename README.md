## Namespace Policy

| Prefix | Namespace | Usage |
|---|---|---|
| `cap:` | `https://hcis.io/ontology/aicapstone/2026/` | Shared AI Capstone course vocabulary. |
| `g13:` | `https://hcis.io/ontology/aicapstone/2026/group13/` | Group 13 individuals, task instances, and affordance individuals. |

Group-specific individuals are declared under `g13:`. Shared course terms such as `cap:Cup`, `cap:Knife`, `cap:ToyBlock`, `cap:TargetObject`, `cap:GraspingAffordance`, and `cap:GraspableObject` are reused from the course vocabulary.

## Ontology Design

The ontology distinguishes four semantic layers:

| Layer | Meaning | Examples |
|---|---|---|
| Object type | Shared class describing what kind of object something is. | `cap:Cup`, `cap:Knife`, `cap:Fork`, `cap:Plate`, `cap:ToyBlock`, `cap:Basket` |
| Task role | Role an object plays in a task. | `cap:TargetObject`, `cap:ReferenceObject`, `cap:CollectableObject`, `cap:ContainerTarget` |
| Affordance | Action possibility associated with an object. | `cap:GraspingAffordance`, `cap:SupportAffordance`, `cap:ContainmentAffordance`, `cap:StackabilityAffordance` |
| Instance | Observed or simulated object individual. | `g13:blueCup01`, `g13:knife01`, `g13:block01` |

## Reasoning Pattern

The ontology defines `cap:GraspableObject` as an OWL equivalent class:

```text
cap:GraspableObject ≡ cap:PhysicalObject
                      AND cap:hasAffordance some cap:GraspingAffordance
```

The intended inference is:

```text
If an individual is a physical object and has at least one grasping affordance,
then a reasoner should classify it as cap:GraspableObject.
```

The following class membership should be inferred rather than manually asserted:

```text
g13:blueCup01  rdf:type  cap:GraspableObject
g13:pinkCup01  rdf:type  cap:GraspableObject
g13:knife01    rdf:type  cap:GraspableObject
g13:fork01     rdf:type  cap:GraspableObject
g13:block01    rdf:type  cap:GraspableObject
```

`g13:plate01` and `g13:basket01` are task-relevant objects but are not expected to be inferred as graspable in this model because they are asserted with support and containment affordances, respectively, not grasping affordances.

## How to Generate Inferred Results

### Option A: Protege

1. Open `ontology/group-ontology.ttl` in Protege.
2. If relative imports are not loaded automatically, manually load:
   - `ontology/imports/course-affordance.ttl`
   - `ontology/imports/course-alignment.ttl`
3. Start an OWL 2 DL reasoner such as HermiT or Pellet.
4. Inspect the inferred members of `cap:GraspableObject`.
5. Export the inferred ontology or inferred axioms as `ontology/inferred-results.ttl`.
6. Save a screenshot of the inferred `cap:GraspableObject` membership under `results/screenshots/` if using the GUI workflow.

### Option B: Jena CLI After Exporting Inferred Results

After `ontology/inferred-results.ttl` has been generated by a reasoner, run:

```bash
arq --data ontology/imports/course-affordance.ttl \
    --data ontology/group-ontology.ttl \
    --data ontology/inferred-results.ttl \
    --query queries/graspable_objects.rq \
    > results/graspable_objects_output.txt
```

To inspect all task objects:

```bash
arq --data ontology/imports/course-affordance.ttl \
    --data ontology/group-ontology.ttl \
    --query queries/task_objects.rq
```

If Apache Jena is not installed, install it first and make sure `riot` and `arq` are available on `PATH`.

## What Is Inferred, Not Asserted

The ontology asserts object types, task roles, and affordance relations. It does not directly assert that the object instances are `cap:GraspableObject`. The `cap:GraspableObject` memberships are expected to be produced by OWL reasoning from:

- object type and physical-object classification from the course ontology;
- asserted `cap:hasAffordance` relations in the group ontology;
- the `cap:GraspableObject` equivalent-class axiom in `ontology/group-ontology.ttl`.

## Limitations

This model treats graspability as a semantic affordance and does not evaluate geometry, mass, collision, gripper aperture, or task success probabilities. The plate and basket are modeled as task-relevant but not graspable under the current task assumptions. A more advanced model could include width, mass, deformability, or gripper-specific constraints.
